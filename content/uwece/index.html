Foundations

Mindset for a Career in Embedded Systems

The question I get most often is: “What should I do to prepare for the career ahead of me?” The answer is usually more about mindset and practice than grades.

Do Not Google the Answers.
Using Google or ChatGPT to chase solutions for these exercises ruins the surprise, takes away the satisfaction of solving the problem, and steals all possible benefit. The world needs better engineers, not better web‑surfers.

Three principles

  1. Stick to your passion and interest and become as proficient and experienced in that line as you can.
  2. Always be working on a project.
    Like EE-design? Design.
    Like embedded software? Get a dev board and make all of the signals renderable and readable — and actually render and read them.
  3. Good grades and strong performance in class matter, but pick your ultimate goal and stay focused on it.

What interviews are really asking

Regardless of how the questions are phrased, most interviews are trying to answer just two things:

  • What is your capacity for knowledge?
  • Do you have passion for the mission?

Capacity for knowledge is about whether you come across as someone who wants to learn new things and is willing to go deeper into every problem rather than skate on the surface.

Passion for the mission is about whether you are just collecting a paycheck or genuinely want to see the project, device, or system launch and succeed.

Would you buy the thing you're being asked to build? Would you recommend it to someone else? If the answer is not obvious, you may be interviewing for the wrong role.

Weekly C Functions

Pick a small, non‑trivial C function every week and re‑implement it from scratch until you don’t have to think about it.

Examples:

  • malloc backed by your own memory pool
  • memcpy that considers overlapping memory

C: Implementing malloc

The signature:

void *malloc(size_t size);

Assume you have a pool:

unsigned char mem[N];

where N is sufficiently large. Your allocator should take from that pool so this usage works: 

unsigned char *p = my_malloc(100);
/* ... */
my_free(p);

C: Implementing memcpy

The signature:

void *memcpy(void *dest, const void *src, size_t n);

Start with the obvious implementation, then reason about performance and overlapping memory regions. Explore what happens when you accidentally implement memmove instead.

Practice Problems

C & C++ Challenges

These aren’t graded assignments. They’re professional warm‑ups: things to write, erase, and write again in different languages until the patterns are part of how you think.

C++: Pointers vs References

In your own words, can you explain the difference between a pointer and a reference in C++? Write down examples that make the semantic differences concrete.

C++: Class Foo

Given this test harness, define and implement a class Foo that makes it work: 

// What must be inside this class definition
// to make the test-case work?

class Foo {
    // ...
};

// Test harness
int main() {
    Foo a("foo");
    Foo b(a);
    std::cout << "a = " << a << std::endl;
    std::cout << "b = " << b << std::endl;

    Foo c("baz");
    std::cout << "c = " << c << std::endl;

    c = b;
    std::cout << "c = " << c << std::endl;

    Foo d = std::move(c);
    std::cout << "c = " << c << std::endl;
    std::cout << "d = " << d << std::endl;

    return 0;
}

The expected output:

a = { 3, "foo"}
b = { 3, "foo"}
c = { 3, "baz"}
c = { 3, "foo"}
c = { 0, "nil"}
d = { 3, "foo"}

Think about ownership, copying, moving, and how you represent the “length” and the “string” inside Foo The main thrust of the question has to do with how the instance of Foo is constructed.

Knight's Tour

Can a Knight start at any arbitrary square on a chessboard and, making only legal moves, visit every other square once and only once?

Write a program that prints an 8×8 board where each square contains the move number when the knight landed there.

  • Only one knight on the board.
  • Start from any location (your algorithm must work from all starts).
  • Visit every square exactly once.
  • Print the board as ASCII.

After you write it in language X, rewrite it in language Y. Do it again later from memory. The point is not the code you keep; it’s the paths your brain builds.
